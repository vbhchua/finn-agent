Why this build

Motivation and takeaways

The question

  • How would a day-one DevRel set up a personal agent to get going — scouring AI events and keynote speakers, and prioritising which events to attend and who to speak with?

Takeaways

  • NemoClaw locks the runtime down well — but customising it is not trivial. Adding the Notion and calendar MCPs meant working with the sealed sandbox, not around it.
  • Model choice is not straightforward either. There are many excellent options, and a real budget buys only a limited number of tests — the host-side endpoint kept each test cheap to set up: swap the model, keep the sandbox.

BLOCKER 1 — collapsed by the upgrade

Pinned cron couldn't drive tools → moved the pin: OpenClaw 2026.6.10 (via a vendored, fail-closed 1-line NemoClaw patch) restored native scheduled tool-calling. All three loops run live in the gateway — heartbeats + digest observed over Telegram.

BLOCKER 2 — retired by measurement

The 120B executor couldn't loop a list → prompts were one item per run (strong model authors, weak model executes). Measured on Kimi K2.6: a 5-event batch ran 85 s (~17 s/event) vs ~57 s for one cold single — 5 distinct research passes, 5 Notion writes, verified by effects. Scaffold dropped 2026-07-10; the loops now drain the due queue. A stronger model made the scaffold unnecessary.

The hard parts

What made it hard — depth of stack

› Tool calls blocked by the sandbox netns

Felt like: the agent reporting "I can't reach your calendar" — while every test we ran said the calendar tool was fine. Why: the sandbox runs each helper tool inside a sealed private network (a netns) with its settings stripped — and our tests ran outside the seal, so they always passed. Fix: the tool reads the network settings off its parent process and relaunches itself with them.

› Health probes that false-negative

Felt like: every standard "is it up?" check said down — while the agent was still replying normally on Telegram. Why: that same sealed network hides the server from outside probes; they can't see in. Fix / lesson: trust the server's own log, never the probe — acting on the false "down" would have restarted a healthy service in a loop.

› New MCP tool not picked up

Felt like: we installed the calendar tool and the config listed it — the agent still said it didn't exist. Why: the agent builds its tool list once and caches it; a settings reload updates the file, not the cache — like an app that won't appear until you reboot. Fix: a full restart of the agent's gateway (and even finding the right process to restart had a trap).

› Personal-account sign-in

Felt like: connecting a personal Outlook calendar should be simple — but the server-style credentials corporate apps use don't exist for personal accounts (no company behind them). Fix: a one-time, phone-style device-code sign-in mints a long-lived token, scoped to the minimum — read-only unless write access is explicitly switched on.

› The model that thought itself to death

Felt like: the weekly digest died with "couldn't generate a response" — after doing all its reading perfectly. Why: the model thinks out loud into the same word budget as its answer — one long think left zero for the reply, and the cut-off bypasses retries. Fix: 4× the output budget — thinking room is answer room.

Five bugs across five layers — sandbox networking, process lifecycle, runtime caching, consumer OAuth, and the model's own output budget. Full technical write-ups: docs/LEARNINGS.md.
Where it goes

Next steps

1
Survive host reboots
A Mac reboot once took the host gateway down for days unnoticed; the host is now an EC2 instance — add boot auto-start (systemd) + an is-finn-alive heartbeat so an outage alerts instead of going unnoticed.
2
Measure cost, not just latency
The batch is measured on latency (~17 s/event vs ~57 s single) and the scaffold is gone — but cost-per-completed-task is still un-measured. Instrument tokens and spend per completed check across executors, so the model choice carries a measured cost.
3
Teach the bootstrap the new schema
The Notion hub grew three pillars (This Quarter · People CRM · Accounts) host-side; extend notion-bootstrap.mjs so a from-scratch rebuild recreates them.
4
Optional hardening
Self-host Firecrawl to regain read-egress control; keep add-ons read-only for higher-trust deployments.
Backup · Trust model

Secure by inheritance — posture ≈ stock NemoClaw

Preserved controls

  • Non-root execution as uid 998
  • Gateway auth via nemoclaw-start
  • Deny-by-default L7 egress proxy — the single authority
  • Runtime-injected secrets — no keys in image layers
  • Telegram pairing — inbound needs manual approval

The deliberate deltas

  • Firecrawl web_fetch: the sandbox still talks to one allowlisted host (api.firecrawl.dev) — Firecrawl's cloud fetches the page on finn's behalf. Net effect: the agent reads any public URL without widening the firewall.
  • Add-ons are read-only by default; writes (calendar, Notion) are an explicit opt-in.
  • Bounded by: cloud fetcher can't reach the sandbox's internal network; read-only, research-only scope.
